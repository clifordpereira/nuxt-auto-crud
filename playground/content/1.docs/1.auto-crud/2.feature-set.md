> The `nuxt-auto-crud stack` automatically reflects Drizzle schemas into a complete RESTful API layer. Using a **Zero-Codegen** approach, the [Core Engine] handles routing and `drizzle-zod` validation without manual boilerplate.

## ðŸ“¡ RESTful Endpoints
The engine automatically mounts the following standard routes for every exported Drizzle table.

| Method | Endpoint | Logic [Core Engine] |
| :--- | :--- | :--- |
| `GET` | `/api/{model}` | List all records with soft-delete filtering. |
| `POST` | `/api/{model}` | Create record with `drizzle-zod` validation. |
| `GET` | `/api/{model}/:id` | Retrieve single record by Primary Key. |
| `PATCH` | `/api/{model}/:id` | Partial update with `drizzle-zod` validation. |
| `DELETE` | `/api/{model}/:id` | Record deletion (Supports soft-delete). |

## ðŸ›¡ï¸ Validation & Protection
The [Core Engine] enforces strict data integrity rules derived from your Drizzle schema.

### System Protected Fields
The following fields are immutable via API requests by default:
- `id` (Primary Key)
- `createdAt` / `created_at`
- `updatedAt` / `updated_at`

### Drizzle-Zod Strategy
Payloads for `POST` and `PATCH` are validated against schemas derived via `createInsertSchema`.
- **POST**: All non-nullable fields required.
- **PATCH**: Partial object verification.

## ðŸŽ¨ Resource Configuration
While zero-config is the default, you can override visibility and updatability in `nuxt.config.ts`.

```typescript
// nuxt.config.ts [Template Implementation]
export default defineNuxtConfig({
  autoCrud: {
    resources: {
      users: ['id', 'name', 'avatar'], // Whitelist for Guest View
    }
  }
})
```

## âš¡ï¸ Key Capabilities
- **Auto-Discovery**: Reflects Drizzle exports into API routes at runtime.
- **Smart Pluralization**: Uses `pluralize` for accurate resource naming (e.g., `user` -> `/api/users`).
- **Coercion**: Automatic type coercion (e.g., Date strings to SQLite timestamps) via `z.coerce`.
